## 1. מה המוצר

**SADAN (סדן)** — מערכת מבוססת בינה מלאכותית לתכנון ותיאום אימונים צבאיים בצה"ל.
המערכת מכסה את **שלב התכנון** — מהנחיות פיקוד ועד תיק תרגיל מאושר.

**השלב הנוכחי:** בניית **דמו מרשים** שיוצג לאלוף "סיסו" (ראש מחלקת אימונים) + אנשי מטה.
הדמו הוא **לב הפרויקט.** אם הוא יצליח — יש המשך פיתוח ותקציב. אם לא — סוף.

**חשוב:** זו לא מערכת ייצור. זה דמו שמדגים יכולות. אבל הוא צריך להיראות ולהרגיש כמו מוצר אמיתי.

### מטרת ה-MVP
בניית דמו פונקציונלי שמדגים את ערך המערכת למחלקת האימונים בצה"ל.
הדמו צריך להראות: המלצת שטחים, בניית תיק תרגיל, ותיאום בין גורמים.

## 4. כללי עבודה — קריטי

### 4.1 עצור ובקש אישור
**כל פעם שיש ספק — עצור ובקש אישור ממור.**
זה כולל:
- החלטות ארכיטקטוניות
- שינויים במבנה תיקיות
- מחיקת קבצים
- הוספת dependencies חדשות
- כל משימה שלא מוגדרת ב-`TASKS.md`
- כל סתירה בין הנחיות

**אל תנחש. אל תחליט לבד על דברים חשובים.**

### 4.2 חלוקת עבודה — אתה ו-Claude (השיחה בדפדפן)
- **Claude בדפדפן (הצ'אט עם מור)** = המוח האסטרטגי. מקבל החלטות, מעצב תסריט, כותב הנחיות
- **אתה (Claude Code)** = המבצע. כותב קוד, עורך קבצים, מריץ שרתים, מדבג
- **מור** = המתאם. מעביר בין שנינו, מאשר החלטות, בודק תוצרים

**אם מור אומר לך "Claude בצ'אט אמר X" — X הוא הנחיה. אל תאתגר.**
**אם אתה רואה בעיה ב-X — עצור ואמור למור "יש בעיה, בבקשה תחזור ל-Claude בצ'אט."**

### 4.3 אל תחרוג מה-scope
- אל תוסיף "ייעול" שלא נתבקש
- אל תשכתב קוד קיים אם הוא עובד
- אל תוסיף dependencies אלא אם חיוני
- אל תיצור קבצים שלא הוגדרו ב-`TASKS.md`

### 4.4 הדמו קודם למוצר
נכון להיום — **כל החלטה תתקבל לפי "מה מרשים בדמו",** לא "מה נכון ארכיטקטונית לטווח ארוך."

**דוגמה:**
- שאלה: "להשתמש בסוכן Claude API אמיתי או בנתונים hardcoded?"
- תשובה: **hardcoded**, אלא אם מור אמר אחרת. כי בדמו חי — הסוכן יכול להחזיר שטות ולהרוס הכל.

**היוצא מן הכלל היחיד:** הווצאפ. ווצאפ חייב להיות **אמיתי** — זה ה-"WOW" של הדמו.

## 9. מודלים מרכזיים (Domain Models)

### TrainingField (שטח אימון)
- `id`, `name`, `location`, `type` (שטח_אש/מתקן/שטח_כינוס)
- `capabilities`: רשימת יכולות (ירי, תרגול_בינוי, וכו')
- `constraints`: מגבלות (תחמוש, שעות, בטיחות)
- `availability`: זמינות

### TrainingTemplate (מתווה אימון)
- `id`, `name`, `description`, `duration_days`
- `objectives`: מטרות האימון
- `required_resources`: משאבים נדרשים
- `unit_type`: סוג יחידה (חי"ר, שריון, וכו')

### DrillFile (תיק תרגיל)
- `id`, `name`, `status` (draft/pending_approval/approved)
- `field_id`, `template_id`
- `schedule`: לו"ז (גאנט)
- `safety_instructions`, `logistics`, `communications`
- `coordination_requests`: בקשות תיאום לגורמים

### CoordinationRequest (בקשת תיאום)
- `id`, `drill_file_id`, `department` (אג"מ/לוגיסטיקה/בטיחות/רפואה/קשר/שטחים)
- `status` (pending/approved/rejected)
- `details`, `response`, `response_date`

### Unit (יחידה)
- `id`, `name`, `type`, `size`, `parent_unit_id`
- `commander_name`, `training_history`

## 10. ארכיטקטורת הדמו — Real vs Fake

הדמו מציג זרימה מלאה של תכנון תרגיל. חלק מהרכיבים **אמיתיים** (עובדים באמת), חלק **מדומים** (hardcoded, נראים אמיתיים).

### Real (עובד באמת):
| רכיב | איך |
|---|---|
| מפה תלת מימדית | MapLibre GL + OpenTopoMap |
| שכבות על המפה | GeoJSON |
| הודעת ווצאפ לרז | whatsapp-web.js → טלפון של רז |
| תגובת רז בווצאפ | server קורא תגובה → מעדכן UI |

### Fake (hardcoded — נראה אמיתי):
| רכיב | למה Fake |
|---|---|
| נתוני שטח (309ה) | נתונים סינתטיים מאומתים |
| 3 מתווי תרגיל | מתווה 1 מתיק המדף, 2+3 סינתטיים — סיכון גבוה מדי להפעיל סוכן חי |
| תיק תרגיל | נכתב מראש, מאומת ע"י רז |
| בוחן 5 שאלות | מוכנות מראש |
| אישור בטיחות (מייל) | סימולציה עם timer |
| אישור קרקע בטוחה | placeholder |
| חסימת כשירות (א'/ב') | לוגיקה ב-frontend |

### החלטה אסטרטגית:
**הסוכנים הקיימים ב-backend (TrainingPlannerAgent וכו') — נשמור לעתיד, אבל לא נפעיל בדמו.**
**לדמו נבנה endpoint פשוט שמחזיר נתונים hardcoded.**

## 11. סוכן הבמ"ל — כיצד עובד (לעתיד)

הסוכן משתמש ב-Claude API ומקבל:
1. הנחיות האימון (מטרות, דגשים, מגבלות)
2. מאגר שטחים וזמינות
3. מאגר מתווים קיימים
4. היסטוריית תרגילים קודמים

הסוכן מחזיר:
- רשימת שטחים מדורגת עם ניקוד והסבר
- המלצה על מתווה מתאים
- טיוטת תיק תרגיל
- רשימת בקשות תיאום נדרשות

**ב-MVP הקיים יש 4 סוכנים:**
- `TrainingPlannerAgent` — מייצר תוכניות אימון
- `ExerciseFileAgent` — מייצר תיק תרגיל
- `CoordinationAgent` — מייצר בקשות תיאום
- `ApprovalTrackerAgent` — מעקב אישורים

**חשוב:** הסוכנים לא כלי עזר לפיתוח — הם **חלק מהמוצר.** אל תמחק אותם.

## 18. אזהרות חשובות

- זה **MVP/דמו** — לא מערכת ייצור
- **נתונים סינתטיים בלבד** — שום מידע מסווג
- הממשק חייב להיות **RTL ובעברית**
- **אין חומרה** — רק תוכנה
- האינטגרציה למערכות צה"ל היא **שלב עתידי**, לא חלק מה-MVP
- **כל הנחה שלא מאומתת = שאלה למור.** אל תמציא פתרונות לשאלות צבאיות.
- **ה-MVP הקיים הוא זהב.** אל תשנה אותו. תעתיק, תשמור, תשתמש.
- **הדמו הוא הלקוח. לא סיסו.** סיסו יראה את הדמו — אבל הלקוח של הקוד שלך הוא הדמו עצמו.
- **עצירות > מהירות.** עדיף לעצור 3 פעמים ולאמת — מאשר לרוץ קדימה ולבנות משהו שגוי.

## 19. הפעלת המערכת ובדיקתה

### הרמת המערכת
```
powershell -ExecutionPolicy Bypass -File start.ps1
```
מעלה את כל השירותים (WhatsApp, Cloudflare tunnel, Backend, Frontend) ממצב לא-ידוע, ומאמת **בפועל** שכל שירות מוכן (לא רק שהפורט פתוח). אם WhatsApp לא מחובר (session ישן/פגום) — מנקה אוטומטית ומנסה שוב. בסיום מציג טבלת סיכום: מה מוכן ✅, ומה דורש פעולה ידנית ⚠️ (כולל הוראות מדויקות, כמו "פתח /qr וסרוק").

### בדיקה לפני דמו
```
powershell -ExecutionPolicy Bypass -File test-system.ps1
```
מניח שהמערכת כבר רצה (אם לא — מפנה ל-`start.ps1`). מבצע בדיקה **אמיתית** מקצה לקצה: שולח הודעת וואטסאפ אמיתית ומבצע שיחת טלפון אמיתית (ל-`TEST_CALL_NUMBER` שב-`backend/.env`) — לא רק בודק שהשרתים עונים. מחזיר ✅ "מוכן לדמו" או רשימת מה נכשל ולמה.

### דילוג בין שלבי הדמו (לעת הצורך מול קהל)
`Ctrl+Shift+→` בכל מסך פותח פאנל נסתר (`DemoControlPanel`) לקפיצה ישירה בין מסכים, מילוי שאלון אוטומטי, ובחירת מתווה — דרך אותם אירועים (`sadan:navigate`/`sadan:action`/`fillField`) שהקול כבר משתמש בהם. לא כולל אישור-הכל באישורים — זה נשאר ידני בכוונה (ה"מומנט" המרכזי של הדמו).

### קובץ עזר משותף
`scripts/sadan-checks.ps1` — פונקציות בדיקה (Backend/Frontend/WhatsApp/Tunnel/Vonage) בשימוש משותף של שני הסקריפטים למעלה.
